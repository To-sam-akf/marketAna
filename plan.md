## pn：01 项目初始化与技术基线
需求：
完成后端、前端、数据库、定时任务和大模型调用的技术选型与工程骨架搭建，保证后续模块可以按流水线方式独立开发、独立测试、统一集成。

核心逻辑：
1. 后端采用 Python 3.11 + FastAPI，按 app/api、app/core、app/models、app/services、app/repositories、app/tasks 分层组织。
2. 前端沿用现有 front_end Vue 3 + Vite 工程，引入 ECharts 用于趋势图、柱状图和统计图。
3. 数据库采用 MySQL，使用 SQLAlchemy 或等价 ORM 管理表结构与 DAO。
4. 配置统一放入 .env，包含数据库连接、大模型 API Key、任务批量大小、规则置信度阈值、轮询间隔等。
5. 建立统一日志、异常码、状态码和接口返回格式，文章处理状态统一使用 0、1、2、3、4、5、-1。

验证方案：
1. 启动后端服务，访问健康检查接口 /health 返回 ok。
2. 启动前端服务，确认首页能正常访问。
3. 本地连接 MySQL 成功，并能执行一次基础表查询。
4. 执行单元测试命令，确认基础工程测试通过。

预期结果：
形成可运行的前后端分离项目骨架，后续各业务模块可在统一结构下开发和联调。

## pn：02 数据库模型与状态流转设计
需求：
建立文章、解析文本、分析结果、任务日志、人工确认记录等核心数据表，支持从原始文章读取到最终预测结果入库的完整状态追踪。

核心逻辑：
1. articles 表保存文章基础信息：id、title、source、company、file_url、file_type、publish_time、status、error_msg、created_at、updated_at。
2. article_texts 表保存解析与清洗结果：article_id、raw_text、cleaned_text、raw_length、cleaned_length、parser_type。
3. analysis_results 表保存趋势分析结果：article_id、product、direction、reason、confidence、analysis_method、need_manual_review、analysis_time。
4. task_logs 表保存调度与流水线执行日志：article_id、stage、status、message、duration_ms。
5. 状态流转规则为：0 未处理 -> 1 解析完成 -> 2 清洗完成 -> 3 规则识别完成 -> 4 LLM 推理完成 -> 5 已入库；任一阶段失败标记 -1。

验证方案：
1. 使用迁移脚本创建所有表，确认字段、索引和外键符合设计。
2. 插入模拟文章，按状态流转更新，确认状态变化可追踪。
3. 对 article_id、status、product、direction、publish_time 建索引后执行查询计划检查。

预期结果：
数据库能够稳定承载文章处理流水线，为任务调度、分析入库、前端筛选和统计查询提供基础。

## pn：03 定时调度模块 Scheduler
需求：
实现每 5 分钟自动扫描待处理文章的能力，按录入时间升序取出固定批次文章并触发后续流水线。

核心逻辑：
1. 使用 APScheduler 或 Crontab 触发任务，默认周期为 5 分钟。
2. 查询 articles where status = 0 order by created_at asc limit N，N 默认 20，可配置。
3. 对命中文章逐条进入 Pipeline：解析 -> 清洗 -> 规则识别 -> LLM 推理或直接入库。
4. 每次扫描写入 task_logs，记录扫描数量、触发数量、成功数量、失败数量。
5. 避免重复执行：处理前将文章加锁或更新为处理中状态，防止并发任务重复消费。

验证方案：
1. 构造 25 条 status = 0 的文章，确认单次只取前 20 条。
2. 模拟空数据场景，确认任务正常退出且记录无命中日志。
3. 并发启动两个调度任务，确认同一文章不会被重复处理。

预期结果：
系统可以自动、稳定、可追踪地拉起文章分析任务，无需人工逐条触发。

## pn：04 文件解析模块 Parser
需求：
支持 PDF、HTML、PNG 或其他图片格式解析，输出可供清洗模块处理的原始文本。

核心逻辑：
1. 根据文件扩展名和 MIME 类型判断解析器类型。
2. PDF 使用 PyMuPDF 按页读取文本，保留页码、标题和段落顺序。
3. HTML 使用 BeautifulSoup 移除 script、style、广告节点，提取 body 正文文本。
4. PNG/JPG 等图片使用 PaddleOCR 或 Tesseract OCR 识别文字。
5. HTML table 或 PDF 表格内容转换为 Markdown 表格，并补充自然语言描述，避免趋势信息丢失。
6. 解析成功后写入 article_texts.raw_text，并更新 articles.status = 1；失败则记录 error_msg 并更新 status = -1。

验证方案：
1. 准备 PDF、HTML、PNG 三类样例文章，分别验证可解析出正文。
2. 准备含表格的样例，确认表格被转换为 Markdown 或结构化文本。
3. 准备损坏文件和不支持格式，确认错误被捕获并写入失败状态。

预期结果：
多格式文章能够统一转换为原始文本，为后续文本清洗和趋势识别提供稳定输入。

## pn：05 数据清洗模块 Cleaner
需求：
清洗解析后的原始文本，去除广告、免责声明、HTML 残留、异常空白和无效噪声，形成统一的 UTF-8 纯文本。

核心逻辑：
1. 使用 chardet 或等价方式检测编码，统一转换为 UTF-8。
2. 维护噪声规则库，过滤“版权所有”“免责声明”“风险提示”“扫码关注”等无关行。
3. 统一换行、空格、全角半角符号和连续空白。
4. 基于文本密度或段落长度保留正文块，剔除导航、页眉、页脚。
5. 记录 raw_length、cleaned_length 和清洗比例，便于发现过度清洗。
6. 清洗成功后更新 article_texts.cleaned_text，并更新 articles.status = 2。

验证方案：
1. 对含广告、免责声明、HTML 标签的样例执行清洗，确认噪声被移除。
2. 对短文本、空文本、乱码文本分别测试异常处理。
3. 检查清洗前后长度差异，避免正文被大面积误删。

预期结果：
得到干净、连续、可分析的文章正文，降低规则识别和 LLM 推理误判率。

## pn：06 规则识别模块 RuleEngine
需求：
优先使用规则引擎识别文章中的明确期货品种、走势方向、理由和置信度，减少不必要的大模型调用成本。

核心逻辑：
1. 建立品种关键词词典，例如螺纹钢、铁矿石、沪铜、原油、豆粕、棕榈油等，并支持同义词和简称。
2. 建立方向规则：看涨、上涨、偏强、反弹、做多归为看涨；看跌、下跌、偏弱、回落、做空归为看跌；震荡、区间、观望归为中性。
3. 使用正则和句子窗口识别“预计.*上涨”“短期.*偏弱”“后市.*震荡”等表达。
4. 理由提取取方向关键词前后各 2 句，保留供前端展示和人工复核。
5. 置信度计算：确定性词汇加分，模糊词汇减分，多处一致结论加分，冲突结论减分。
6. 置信度 >= 0.7 时直接生成分析结果；置信度 < 0.7 时标记进入 LLMInfer。
7. 规则识别完成后更新 articles.status = 3。

验证方案：
1. 编写覆盖看涨、看跌、中性、模糊观点、多品种观点的单元测试。
2. 对明确结论样例验证能直接入库，且 analysis_method = rule。
3. 对模糊样例验证会进入 LLM 推理路径。

预期结果：
明确观点文章可快速、低成本完成结构化分析，模糊文章被准确分流到大模型模块。

## pn：07 LLM 推理模块 LLMInfer
需求：
对规则无法高置信识别的文章调用大语言模型，输出品种、方向、理由和置信度，并对低置信结果标记人工确认。

核心逻辑：
1. 输入 cleaned_text、title、source、company、publish_time 等上下文。
2. Prompt 设定系统角色为“资深期货市场分析师”，要求从文章中提取期货品种和后续走势预测。
3. 输出强制为 JSON：{"product":"","direction":"看涨/看跌/中性","reason":"","confidence":0.0}。
4. 对模型返回进行 JSON 解析、字段校验、枚举值校验和置信度范围校验。
5. 置信度 < 0.5 时设置 need_manual_review = true，并在前端高亮“待人工确认”。
6. 记录模型名称、调用耗时、错误信息和重试次数。
7. 推理成功后更新 articles.status = 4，并交由 Repository 入库。

验证方案：
1. 使用模拟 LLM 返回测试 JSON 解析成功、字段缺失、非法方向、非法置信度等场景。
2. 使用真实或测试模型对 5 篇模糊文章试跑，检查输出是否符合结构。
3. 模拟 API 超时和失败，验证重试、失败记录和状态回滚策略。

预期结果：
隐含观点和模糊表达文章也能被结构化分析，且低置信结果可进入人工复核流程。

## pn：08 数据存储模块 Repository
需求：
封装文章读取、文本保存、分析结果保存、状态更新和统计查询接口，保证业务模块不直接操作 SQL 细节。

核心逻辑：
1. 提供 get_pending_articles(limit)、save_raw_text、save_cleaned_text、save_analysis_result、update_status、save_task_log 等 DAO 方法。
2. 分析结果入库时保证幂等，同一 article_id 重跑时更新旧结果或写入版本号，避免重复统计。
3. 对关键写入操作使用事务，确保结果、状态、日志一致。
4. 提供前端统计查询：今日文章数、各方向数量、品种趋势、期货公司筛选、时间范围筛选。
5. 对失败文章保存 stage、error_msg 和可重试标记。

验证方案：
1. 编写 Repository 单元测试，覆盖新增、更新、查询、事务回滚。
2. 重复保存同一文章分析结果，确认不会产生重复有效记录。
3. 构造失败场景，确认状态和日志同时写入。

预期结果：
数据访问层稳定可靠，支撑流水线执行和前端展示，降低模块之间耦合。

## pn：09 后端 API 模块
需求：
为前端提供统计总览、筛选查询、趋势图、文章列表、文章详情、手动刷新和人工确认相关 RESTful API。

核心逻辑：
1. GET /api/dashboard/summary 返回今日分析统计、成功率、文章数量、待人工确认数量。
2. GET /api/articles 支持按品种、期货公司、预测方向、时间范围、关键词分页查询。
3. GET /api/articles/{id} 返回文章详情、清洗文本、预测结果、理由和处理日志。
4. GET /api/trends 返回按品种和时间聚合的看涨、看跌、中性趋势数据。
5. POST /api/tasks/run 手动触发一次扫描或指定文章重跑。
6. POST /api/results/{id}/confirm 支持人工修正确认品种、方向、理由和置信度。
7. 所有接口返回统一 JSON 格式，错误包含 code、message、detail。

验证方案：
1. 使用 pytest 或接口测试工具覆盖正常查询、空结果、非法参数、分页边界。
2. 使用模拟数据验证 dashboard 和 trends 聚合结果正确。
3. 联调前端，确认筛选参数和返回字段一致。

预期结果：
后端 API 能完整支撑前端页面和人工复核工作流，并可单独测试与排错。

## pn：11 流水线联调与异常处理
需求：
把 Scheduler、Parser、Cleaner、RuleEngine、LLMInfer、Repository 串成完整自动化流水线，并补齐异常重试、失败标记和日志审计。

核心逻辑：
1. Pipeline 入口接收 article_id，按状态执行对应阶段，支持从失败阶段重试。
2. 每个阶段开始、成功、失败都写入 task_logs。
3. 网络、文件读取、OCR、LLM 调用等易失败环节设置超时和有限重试。
4. 对失败文章更新 status = -1，同时保留 error_msg 和失败阶段。
5. 对大批量任务限制并发数，避免数据库和模型 API 压力过高。
6. 增加处理耗时统计，用于后续性能优化。

验证方案：
1. 使用 20 篇混合格式样例文章跑完整流水线，确认状态最终到 5 或 -1。
2. 人为制造解析失败、清洗失败、LLM 超时，确认日志和状态符合预期。
3. 对同一批数据重跑，确认幂等和重复消费控制有效。

预期结果：
系统能够从数据库自动读取文章并完成解析、清洗、分析、推理、入库的端到端处理。
